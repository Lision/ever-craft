# 微信公众号贴图生成 Skill 设计

## 背景与目标

创建一个 Codex skill，将完整文章或结构化大纲转换为一组可直接发布的微信公众号贴图卡片。图片承担主要信息，发布文案只作辅助说明。

成品采用文章式组织：一张主图表达整篇内容的核心命题，每个主要章节对应一张副图。整组图片保持统一的配色、排版、线条、角色和信息层级，同时允许每页使用独立的视觉隐喻。

首版以 skill 验证工作流能否稳定兜住效果；如果仅靠 skill 无法达到可接受的一致性和质量，再考虑实现更强的独立 workflow。

## 范围

### 包含

- 接受完整文章或结构化大纲。
- 自动拆分主图、章节副图和可选总结页。
- 允许用户覆盖标题、页数、章节映射和重点。
- 生成原创的 Notion 式极简线稿插图。
- 使用确定性排版合成准确的中文文字。
- 输出 1080×1440 PNG 完整卡片。
- 使用独立 sub-agent 检查成品，并以结构化 review 驱动局部重做。
- 将源文、状态、视觉规范、素材、成品和 review 保存到 Git 工程的单篇子目录。

### 不包含

- 自动发布到微信公众号。
- 手机缩略预览或微信压缩模拟。
- 复刻参考作品的具体角色、构图、品牌标识或单张插图。
- 无限自动迭代。
- 在公开 skill 仓库中捆绑字体文件。

## 输入模式

Skill 支持两种等价入口：

1. 完整文章：自动提取主题、核心判断、章节和结论。
2. 结构化大纲：接收主标题、副标题、章节标题、章节正文和强调内容。

用户可以覆盖自动结果，包括：

- 主标题和副标题。
- 必须保留或删除的章节。
- 页面数量。
- 章节与页面的映射。
- 每页必须保留的句子。
- 是否使用固定角色。
- 账号署名。
- 输出 Git 工程目录。

## 角色与职责

### 主 agent

- 解析输入并维护生成工程。
- 生成卡片脚本和 `manifest.yaml`。
- 维护 `visual-bible.yaml`。
- 协调用户确认、生图、排版和 review。
- 根据 review 的 owner 和依赖关系路由返工。
- 不能用自查替代独立 review。

### 生图 sub-agent

- 根据单页 brief、视觉规范和锚点图生成无排版文字的插图层。
- 只处理被分配的图片问题。
- 不评价自己的结果，也不能自行宣布通过。

### Review sub-agent

- 独立检查单页和整组成品。
- 不复用生图 sub-agent 的判断。
- 输出原子化、可路由、可复查的结构化问题。
- 不能直接修改源文、manifest、插图或卡片。

## 三个确认门禁

### Gate 1：卡片脚本

正式生图前，向用户确认：

- 核心命题。
- 页面数量和顺序。
- 每页的唯一中心观点。
- 主图和副图文案。
- 页面类型和视觉隐喻。

默认生成一张主图和三至八张副图。每个主要章节对应一张副图；内容过密时拆页，不通过缩小字号硬塞。

### Gate 2：视觉锚点

先生成并确认：

- `style-anchor.png`：整组的线条、色彩、留白、复杂度和质感基准。
- `character-sheet.png`：启用角色时的轮廓、比例、眼睛、动作和视角基准。

用户确认后才批量生成副图。关闭角色时不创建 `character-sheet.png`。

### Gate 3：最终交付

全部页面完成独立 review 后再交付。Review 判定通过不取代用户的最终确认。

## 内容编排

每页只表达一个中心观点。卡片脚本为每页定义：

- 页面 ID 和顺序。
- 页面类型：封面、标准章节、对比或列表、总结。
- 标题、副标题、正文和强调内容。
- 必须保留的文字与可压缩内容。
- 视觉隐喻。
- 插图 brief。
- 与其他页面的内容或视觉依赖。

页面职责：

- 主图：建立核心命题，标题最大，信息最少，视觉隐喻最强。
- 标准章节页：解释一个章节观点，以插图和有限信息块支撑理解。
- 对比或列表页：只在内容天然具有并列、比较或分类结构时使用。
- 总结页：回收核心结论，可加入行动建议或互动问题，不重复整篇文章。

## 视觉系统

### 固定骨架

- 画布：1080×1440，3:4 竖图。
- 色板固定使用从用户参考图解码像素中提取的值，不使用主观颜色名：

  | Token | 色值 | 用途 |
  | --- | --- | --- |
  | `background` | `#FAFAF8` | 页面暖白背景 |
  | `surface` | `#F0F0EE` | 信息块和浅灰分区 |
  | `ink` | `#0A0A08` | 标题、正文和主要线稿 |
  | `solid` | `#000000` | 角色实体和最高对比元素 |
  | `accent` | `#012FA7` | 蓝色标题、标签和重点信息块 |
  | `annotation` | `#854953` | 手绘下划线、箭头和少量批注 |
  | `muted` | `#747472` | 页眉、英文标签和次要信息 |
  | `divider` | `#D1D1CF` | 分隔线和弱边界 |

- 上述颜色是本 skill 的默认确定值；用户明确要求换主题时才允许覆盖，并把覆盖值写入当次 `visual-bible.yaml`。
- 字体：本地安装的 `Maple Mono NF CN`，无静默 fallback。
- 布局：大面积留白、细分隔线、明确的信息层级和稳定网格。
- 插图：原创、简易、不规则的手绘线条；避免复杂光影、3D、摄影感和装饰堆叠。

### 角色系统

- 默认使用原创极简抽象角色，作为观察者、推动者或问题承受者。
- 固定角色轮廓、比例、眼睛和线条特征，动作随章节变化。
- 用户可关闭角色，改用物体隐喻、空间关系或抽象图解。
- 不照搬参考案例中的具体黑色角色。

### 原创边界

外部参考只用于提取高层特征，例如留白、线稿、编辑设计和信息层级。不得复刻参考作品的具体角色、构图、品牌署名或单张插图。Review 必须检查结果是否与参考案例过于接近。

## 生成与合成

### 1. 内容规划

将用户确认的卡片脚本写入 `manifest.yaml`。Manifest 是生成任务的唯一事实来源。

### 2. 插图生成

- 只生成无排版文字的插图层。
- 必要标签尽量在确定性排版阶段添加，避免模型生成错误中文。
- 每次生图都提供单页 brief、`visual-bible.yaml`、`style-anchor.png` 和可选 `character-sheet.png`。
- 后续轮次同时提供当前图片和对应 review action。

### 3. 确定性排版

配套渲染脚本将标题、正文、标签、页码、署名和插图合成为 PNG。

渲染前必须验证：

- `Maple Mono NF CN` 能在本机被定位。
- 字体覆盖本次需要的中文字符和标点。
- 文本没有溢出或落入安全边距之外。

字体无法定位或字符不完整时明确报错，不自动换字体。

### 4. 独立 Review

每次生图和合成后，必须由独立 Review sub-agent 检查。Review sub-agent 接收：

- 原始文章或大纲。
- 用户确认的卡片脚本。
- `manifest.yaml`。
- `visual-bible.yaml`。
- 锚点图。
- 全部当前成品卡片。
- 上一轮 review 记录，仅第二轮及以后提供。

## Review 合同

### 单页检查

- 观点是否准确、独立且易懂。
- 插图隐喻是否对应文字。
- 标题层级、字号和留白是否合理。
- 是否存在错字、溢出、低对比度或视觉噪音。
- 主图是否表达核心命题并具有吸引力。

### 整组检查

- 主图与副图是否属于同一视觉系统。
- 角色、配色、线条和构图语言是否稳定。
- 阅读顺序和章节递进是否自然。
- 相邻页面是否过度重复。
- 信息密度是否均衡。
- 是否存在单页好看但整组松散的问题。
- 是否与外部参考过于接近。

### 原子问题

同一页面可以有多个问题，但每条 issue 只能有一个主要 owner：`content`、`image`、`layout` 或 `system`。有顺序关系时使用 `depends_on`。

```yaml
verdict: revise
global_issues: []
pages:
  - page: p03
    issues:
      - id: p3-content-01
        severity: major
        owner: content
        issue: 核心结论没有说明权限边界
        action: 将副标题改为明确的判断句

      - id: p3-image-01
        severity: major
        owner: image
        issue: 插图只表达连接，没有表达受控访问
        action: 保留角色，增加闸门和权限凭证隐喻
        depends_on:
          - p3-content-01

      - id: p3-layout-01
        severity: minor
        owner: layout
        issue: 底部信息块文字过密
        action: 合并信息块并增加上下留白
```

复审必须按 issue ID 标记 `resolved`、`partially_resolved` 或 `unresolved`。

## Review 驱动的局部返工

- `content`：主 agent 先修改文案，再判断是否影响插图和排版。
- `image`：将原 brief、当前图片和 review action 交回生图 sub-agent。
- `layout`：保留插图，只重新渲染。
- `system`：更新 `visual-bible.yaml`，标记并重新处理全部受影响页面。

处理顺序默认是 `content`、`image`、`layout`。跨页问题和视觉规范问题优先于单页局部问题。

## 内容变更的影响传播

- 只改措辞：重新排版。
- 改变视觉对象或关系：重新生图并排版。
- 改变文章核心命题：主图和相关副图失效，退回 Gate 1。
- 改变视觉系统：退回 Gate 2，并重新处理所有受影响页面。

主 agent 必须在 `manifest.yaml` 中记录被失效的页面和原因，不能只修 review 直接点名的页面。

## 轮次、通过标准与停止条件

- 整组最多三轮生图。
- 任意单页最多生成三次。
- 初次生图计为第一次。
- 排版重新渲染不计入生图轮次。
- 每次生图后必须再次经过独立 review。
- `critical` 或 `major` 问题未解决时禁止标记通过。
- 仅剩不影响阅读和一致性的 `minor` 建议时，reviewer 可以判定通过并保留记录。
- 同一 issue 连续两轮未解决时停止自动重试，即使尚未达到三次上限。
- 达到整组或单页上限后停止自动生图。
- 停止时交付当前最佳版本、未解决问题和具体能力限制，不能声称通过。

## Git 工程目录与交付

用户指定一个 Git 工程作为生成根目录。每篇贴图使用一个 `<post-slug>` 子目录：

```text
<git-project>/<post-slug>/
├── source.md
├── manifest.yaml
├── visual-bible.yaml
├── style-anchor.png
├── character-sheet.png
├── illustrations/
├── cards/
└── reviews/
    ├── round-01.yaml
    ├── round-02.yaml
    └── final.yaml
```

### 文件职责

- `source.md`：保存原始文章或大纲、用户指定内容和参考链接；不保存后续压缩文案。
- `manifest.yaml`：保存卡片脚本、prompt、当前文件路径、状态、依赖、失效原因和轮次；是唯一事实来源。
- `visual-bible.yaml`：保存画布、字体、上述确定色板、网格、线条、角色、署名和禁用元素。
- `style-anchor.png`：保存整组风格基准。
- `character-sheet.png`：保存可选角色基准；角色关闭时不存在。
- `illustrations/`：保存每页无文字插图及其历史版本，例如 `p03-v02.png`。
- `cards/`：只保存当前有效的完整 PNG，例如 `p03.png`；排版历史不保留。
- `reviews/round-NN.yaml`：不可覆盖的逐轮独立 review。
- `reviews/final.yaml`：保存最终 verdict、剩余问题、轮次和停止原因。

最终交付以该子目录为准，不再创建额外的交付包。

## Skill 文件结构

```text
skills/generating-wechat-cards/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   ├── visual-system.md
│   └── content-schema.md
├── scripts/
│   ├── render_cards.py
│   └── validate_manifest.py
└── assets/
    └── default-theme.yaml
```

- `SKILL.md`：描述触发条件、门禁、角色职责、生成流程、review 路由和停止条件。
- `references/visual-system.md`：保存详细视觉规则、页面类型和插图提示词约束。
- `references/content-schema.md`：保存 manifest、visual bible 和 review schema。
- `scripts/render_cards.py`：使用 `Maple Mono NF CN` 确定性合成卡片。
- `scripts/validate_manifest.py`：在生图和渲染前验证字段、文件、依赖、状态和轮次。
- `assets/default-theme.yaml`：保存可覆盖的默认尺寸、色板和排版参数。

## 验证策略

Skill 采用面向流程文档的 RED-GREEN-REFACTOR：

1. 在没有新 skill 的情况下运行真实任务基线，记录 agent 在内容拆页、文字生图、跨页一致性、review 路由和停止条件上的失败。
2. 创建最小 skill 和资源，只处理基线暴露的问题。
3. 使用同类输入重新运行，验证 agent 能否遵守门禁、生成工程、调用独立 sub-agent、执行原子 review 和局部返工。
4. 实际运行渲染脚本，验证字体定位、文字布局、PNG 尺寸和错误处理。
5. 使用一篇真实文章完成端到端试跑，并根据产物判断 skill 是否足够；若无法稳定兜住关键效果，再设计独立 workflow。

## 成功标准

- 用户可用完整文章或大纲启动任务。
- 正式批量生图前完成卡片脚本和视觉锚点确认。
- 中文由确定性排版生成，无模型乱码或错字。
- 主图准确表达核心命题，副图各自表达一个章节观点。
- 整组在配色、线条、角色和排版上保持可辨识的一致性。
- 每轮均有独立、结构化、原子化的 sub-agent review。
- Review 建议能被准确路由到内容、生图、排版或视觉系统。
- 自动流程遵守三轮上限和连续两轮未解决停止条件。
- 所有输入、状态、产物和 review 都保存在单篇 Git 子目录中。
